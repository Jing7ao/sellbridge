关键发现：东南亚跨境 SaaS 赛道的最大竞品是 店小秘（全、大、但旧）和 BigSeller（免费、深耕印尼）。但两者都不是 AI 原生产品，UI 停留在 5 年前的水平。SellBridge 的窗口：AI 原生 × 极致体验 × 轻量定价 × 独立开发者速度。

三、SellBridge 的核心价值再思考

"如果平台本身可以上架商品，为什么还需要 SellBridge？"

这是一个产品存在价值的根本性问题。答案是：

场景直接在平台操作用 SellBridge
在 4 个平台上架 1 个商品 登录 4 次，手动填 4 次表单，每次用不同语言 输入一次中文，60 秒完成 4 平台上架
在 4 平台 × 6 市场上架 理论上 24 次操作，实际不可能（语言障碍） 一次操作，AI 自动翻译 6 种语言 + 本地化 SEO
查看 4 个平台的订单 每天登录 4 个后台，手动汇总 一个页面看所有平台订单
跨平台比价 手动切换 4 个 App / 网页，Excel 记录 自动聚合、价格变化趋势图、告警
库存跨平台同步 A 平台卖出一件，手动去 B/C/D 平台改库存 自动同步，避免超卖

SellBridge 的价值不是"替代平台功能"，而是"跨平台的聚合层"。单个平台的后台是完备的，但 多个平台之间的缝隙——翻译、数据同步、库存协调、订单汇总——才是 SellBridge 存在的理由。

4.2 产品差异化路线

  1. AI 翻译做到极致——让翻译结果不像翻译，像本地人写的。针对泰国/越南/印尼/马来/新加坡市场分别微调 prompt，加入当地电商文案特色（泰国用 emoji + 促销语气、越南用功能罗列、新加坡用简洁商务风）。这是大厂做不到的精细度。
  2. "60 秒上架"作为传播钩子——把这个体验做成可传播的视频："一个中国卖家如何在 60 秒内把商品铺到东南亚 5 国 4 个平台"。在 TikTok / 小红书 / Twitter 上发布。
  3. 开源平台 Adapter 层——把 src/adapters/ 下的 Lazada/Shopee/TikTok Shop API 封装库开源（MIT 协议）。这会吸引开发者流量，建立技术信任，同时不威胁核心商业价值（商业价值在翻译引擎和业务逻辑层）。
  4. 不做全链路，做"跨平台中枢"——不碰 WMS（仓储）、TMS（物流）、财务等重模块。聚焦：刊登 / 订单汇总 / 价格监控 / 库存同步 / AI 客服。五个模块形成"跨平台经营必需"的闭环。

六、功能拓展：订单管理 + 库存管理

6.1 为什么不矛盾

平台上架商品 ≠ 跨平台管理。前者是"创建"，后者是"聚合"。

SellBridge 新增订单管理的价值：一个卖家在 Lazada TH、Shopee TH、TikTok Shop TH、Shopify 同时卖货 → 每天登录 4 个后台查订单 → SellBridge 一个页面汇总所有平台订单，按状态筛选，批量标记发货。这不是替代平台后台，而是避免 4 个后台之间来回切换。

同理库存管理：A 平台卖出一件 → 手动去 B/C/D 平台改库存 → SellBridge 自动同步库存数据，低库存告警。超卖是跨境卖家的核心痛点之一。

6.2 订单管理模块设计（轻量版）

功能说明技术依赖
订单汇总从已连接店铺拉取订单列表，按平台分组显示。支持按状态（待发货/已发货/已取消/退货）筛选各平台 Adapter 新增 Order API
订单详情点击展开：商品信息、收件地址、金额、物流单号—
状态更新标记发货、输入物流单号（单向同步到平台）平台发货 API
统计概览今日订单数、待处理数、各平台订单占比饼图Recharts

不做的：退款/纠纷处理（太重，需人工介入）、物流轨迹追踪（可外链到平台物流页面）、发票管理。

6.3 库存管理模块设计（轻量版）

功能说明技术依赖
库存总览按商品/SKU 展示各平台库存数量，跨平台对比各平台商品库存 API
低库存预警可设置阈值（如 < 10 件），低于阈值时页面标记 + 推送通知—
一键同步手动触发：将某个商品的库存数同步到所有平台平台库存更新 API
自动同步（未来）A 平台产生订单 → 自动扣减 B/C/D 平台库存 → 触发低库存预警Webhook + 订单-库存联动

不做的（现阶段）：入库/出库单据管理（WMS 范畴）、多仓库管理、批次/保质期管理。
